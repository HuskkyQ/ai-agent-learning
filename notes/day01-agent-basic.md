# Day 01: 为什么 Agent 不等于普通聊天机器人？

## 原回答简评

我的原回答方向是对的：普通聊天机器人主要是用户驱动的一问一答；Agent 在此基础上增加了目标导向、任务执行、工具使用、结果检查和失败退出能力。

需要补充的地方：

- 需要更明确地区分 chatbot、workflow、agent、multi-agent。
- 需要解释 Day 1 验收标准里的 `observe -> think -> act -> observe` 循环。
- 需要说明不是所有多轮 LLM 应用都应该叫 Agent。

## 更完整的回答

Agent 不等于普通聊天机器人，因为二者的核心职责不同。

普通聊天机器人主要负责“对话”。用户提出问题，模型根据已有上下文生成回答。这个过程通常由用户推动：用户问，模型答；用户继续问，模型继续答。即使它能连续多轮对话，本质上也主要是在生成文本回复。

Agent 更接近“目标驱动的任务执行系统”。用户给它一个目标后，它不仅要回答问题，还要围绕目标主动规划下一步，必要时调用工具、读取外部信息、执行动作、检查结果，并根据反馈调整自己的行为。

一个典型 Agent 会进入这样的循环：

1. **Observe**：观察当前任务、用户输入、工具返回结果或环境状态。
2. **Think**：判断当前状态，决定下一步应该做什么。
3. **Act**：调用工具、查询资料、写文件、执行 API 请求，或向用户提问。
4. **Observe**：读取动作结果，再决定是否继续、修正、停止或交还控制权。

所以 Agent 的关键不只是“会聊天”，而是能把语言理解、决策、工具调用和反馈循环组合起来完成任务。

## Chatbot、Workflow、Agent、Multi-agent 的区别

| 类型 | 核心特征 | 谁控制流程 | 例子 |
| --- | --- | --- | --- |
| Chatbot | 主要生成回复 | 用户主导 | 问答助手、客服聊天窗口 |
| Workflow | 按固定步骤执行 | 开发者写好的代码路径 | 先分类、再检索、再总结 |
| Agent | 根据状态动态决定下一步 | LLM 参与控制流程 | Coding agent、网页操作助手 |
| Multi-agent | 多个 Agent 分工协作 | 多个 Agent 或 manager 协调 | 一个 Agent 查资料，另一个 Agent 写报告 |

Workflow 和 Agent 的边界尤其重要。一个系统只要用了 LLM 和多个步骤，并不一定就是 Agent。如果每一步都由代码固定写死，它更像 workflow。只有当模型能根据任务状态动态选择步骤、工具和停止条件时，才更接近 Agent。

## Agent 的能力边界

一个更完整的 Agent 通常需要具备这些能力：

- **理解目标**：知道用户真正要完成什么，而不只是回答一句话。
- **规划步骤**：能把目标拆成多个可执行动作。
- **使用工具**：能调用搜索、数据库、文件、API、代码执行器等外部工具。
- **读取反馈**：能根据工具结果判断当前方案是否有效。
- **修正行为**：失败时能调整方案，而不是一直重复同一个错误。
- **判断停止**：能知道任务何时完成，或何时应该停止并交还给用户。
- **遵守边界**：在权限、安全、成本和风险限制内行动。

## 什么时候不要用 Agent？

Agent 不是越复杂越好。很多任务用普通 chatbot、单次 LLM 调用或固定 workflow 就够了。

不适合 Agent 的情况：

- 任务很简单，一次回答就能解决。
- 流程稳定，固定规则足够可靠。
- 成本和延迟很敏感，不适合多轮调用。
- 工具调用风险高，但没有权限控制、日志、人工确认或失败退出机制。

适合 Agent 的情况：

- 任务开放，无法提前写死所有步骤。
- 需要多轮观察、判断、执行和修正。
- 需要调用外部工具或系统完成真实动作。
- 任务结果可以通过测试、日志、用户反馈或评估集检查。

## 我的最终理解

普通聊天机器人主要是在“回答”；Agent 是在“执行任务”。

Agent 的本质是一个围绕目标运行的反馈循环：它观察当前状态，思考下一步，执行动作，再观察动作结果。这个循环让 Agent 能够处理比普通问答更复杂的任务，但也带来更高成本、更高延迟和更高风险。因此，真正的 Agent 开发不只是写 prompt，还要设计工具、流程、日志、评估、安全边界和失败退出机制。

所以，Agent 不是普通聊天机器人的换皮版本，而是把 LLM 放进一个可执行、可反馈、可约束的任务系统里。
