## 引言

大语言模型正在越来越擅长处理复杂的多步骤任务。随着推理、多模态和工具使用能力的发展，一类由 LLM 驱动的新系统开始成熟：Agent。

这份指南面向正在探索第一个 Agent 的产品和工程团队。它把客户部署经验整理成实践建议，包括如何识别适合 Agent 的用例、如何设计 Agent 的逻辑和编排方式，以及如何让 Agent 在生产环境中更安全、更可预测、更有效。

读完之后，你应该能建立构建第一个 Agent 所需的基础判断框架。

## 什么是 Agent？

传统软件可以帮助用户简化或自动化工作流；Agent 则更进一步，可以代表用户以较高独立性完成同类工作流。

可以把 Agent 理解为：**能够代表你独立完成任务的系统**。

这里的 workflow 指为了达成用户目标而必须执行的一系列步骤。例如：

- 解决客服问题。
- 预订餐厅。
- 提交代码变更。
- 生成报告。

并非所有接入 LLM 的应用都是 Agent。简单聊天机器人、单轮 LLM 调用、情感分类器这类应用，如果没有让 LLM 控制工作流执行，就不属于 Agent。

一个可靠 Agent 通常具备两个核心特征：

1. **由 LLM 管理工作流执行并做决策**：它能判断任务是否完成，必要时修正自己的行动；失败时也能停止执行，并把控制权交还给用户。
2. **能使用工具与外部系统交互**：它可以获取上下文，也可以执行动作；会根据当前工作流状态动态选择工具，并始终在明确护栏内运行。

## 什么时候应该构建 Agent？

构建 Agent 需要重新思考系统如何做决策、如何处理复杂性。和传统自动化相比，Agent 更适合传统确定性规则难以覆盖的工作流。

以支付欺诈分析为例：传统规则引擎像检查清单，根据预设条件标记交易；LLM Agent 更像有经验的调查员，会结合上下文、识别微妙模式，并在规则没有明确命中的情况下发现可疑活动。

评估 Agent 用例时，优先考虑过去难以自动化、传统方法摩擦较大的流程：

- **复杂决策**：涉及细腻判断、例外情况或上下文敏感决策，例如客服中的退款审批。
- **难维护规则**：规则数量庞大且复杂，更新成本高、容易出错，例如供应商安全审查。
- **高度依赖非结构化数据**：需要理解自然语言、解析文档、从对话中提取含义，例如处理家庭保险理赔。

在投入构建 Agent 前，应验证用例是否清楚满足这些条件。否则，确定性方案可能已经足够。

## Agent 设计基础

最基础的 Agent 由三个组件构成：

1. **模型**：负责推理和决策的 LLM。
2. **工具**：Agent 可调用的外部函数或 API。
3. **指令**：定义 Agent 行为方式的明确指南和护栏。

原文随后用 OpenAI Agents SDK 展示了一个天气 Agent 示例：给 Agent 配置名称、说明和天气查询工具。相同概念也可以用其他库实现，或者直接从零构建。

```
# 示例
weather_agent = Agent(
    name="Weather agent",
    instructions="You are a helpful agent who can talk to users about the weather",
    tools=[get_weather],
)
```

### 选择模型

不同模型在任务复杂度、延迟、成本方面有不同取舍。一个工作流中可以针对不同任务使用不同模型。

并不是每个任务都需要最强模型。简单检索或意图分类可以由更小、更快的模型处理；像“是否批准退款”这类更复杂决策，可能需要更强模型。

推荐做法：

1. 先用最强模型为所有任务建立原型，得到性能基线。
2. 再逐步替换为更小模型，观察是否仍能满足要求。
3. 这样可以避免一开始就限制 Agent 能力，也能看清小模型在哪些环节可用、在哪些环节失败。

选择模型的原则：

1. 建立 evals，用评估确定性能基线。
2. 先用可用的最好模型达到准确率目标。
3. 在可接受范围内，用更小模型优化成本和延迟。

### 定义工具

工具通过底层应用或系统的 API 扩展 Agent 能力。对于没有 API 的旧系统，Agent 也可以依靠 computer-use 模型像人类一样操作网页或应用界面。

每个工具都应有标准化定义，让工具和 Agent 之间形成灵活的多对多关系。工具越清晰、测试越充分、复用性越好，就越容易发现、维护和管理版本，也能减少重复定义。

Agent 通常需要三类工具：

| 类型     | 作用                               | 示例                                         |
| -------- | ---------------------------------- | -------------------------------------------- |
| 数据工具 | 获取执行工作流所需的上下文和信息   | 查询交易数据库、CRM、读取 PDF、搜索网页      |
| 动作工具 | 与系统交互并执行动作               | 发送邮件或短信、更新 CRM、把客服工单转给人工 |
| 编排工具 | 让 Agent 本身成为其他 Agent 的工具 | 退款 Agent、研究 Agent、写作 Agent           |

### 配置指令

高质量指令对任何 LLM 应用都重要，对 Agent 尤其关键。清晰指令可以减少歧义、改善 Agent 决策，并降低工作流执行错误。

#### Agent 指令最佳实践

- **使用现有文档**：将操作规程、支持脚本、政策文档转成 LLM 友好的 routine。客服场景中，一个 routine 可以对应知识库中的一篇文章。
- **让 Agent 拆解任务**：把密集材料拆成更小、更清楚的步骤，能减少歧义，让模型更容易遵循。
- **定义清楚动作**：每一步都应对应明确动作或输出，例如向用户询问订单号，或调用 API 查询账户详情。
- **覆盖边界情况**：真实交互会出现信息不完整、问题超出预期等情况。稳健的 routine 要提前设计条件分支和备用步骤。

### 编排

有了模型、工具、指令之后，就可以考虑 Agent 如何执行工作流。

虽然一开始就构建复杂的全自治 Agent 很诱人，但 OpenAI 的客户实践显示，增量式方法通常更容易成功。

编排模式大体分为两类：

1. **单 Agent 系统**：一个模型配合合适工具和指令，在循环中执行工作流。
2. **多 Agent 系统**：多个协调的 Agent 共同分担工作流执行。

#### 单 Agent 系统

单个 Agent 可以通过逐步增加工具来处理很多任务。这样复杂度更可控，评估和维护也更简单。每增加一个工具，都能扩展能力，而不必过早引入多 Agent 编排。

> 图示位置：原文此处展示单 Agent 工作流：Input -> Agent -> Output，并在 Agent 下方标出 Instructions、Tools、Guardrails、Hooks 等层。

任何编排方式都需要一个“run”的概念，通常表现为一个循环：Agent 不断运行，直到触发退出条件。

常见退出条件包括：

- 调用了特定工具。
- 生成了某种结构化输出。
- 出现错误。
- 达到最大轮数。

原文以 Agents SDK 为例说明：Agent 会持续循环，直到调用 final-output 工具，或模型返回不含工具调用的响应。

> 代码示例位置：原文此处展示 `Agents.run(...)` 的调用方式。

这个 while loop 是 Agent 工作机制的核心。多 Agent 系统中，也可以发生一连串工具调用和 Agent handoff，但模型仍是在退出条件前持续运行多步。

在不切换到多 Agent 框架的情况下，管理复杂度的有效策略是使用 prompt templates。与其维护大量单独 prompt，不如使用一个灵活基础 prompt，并通过变量注入不同政策或上下文。

##### 什么时候考虑创建多个 Agent？

OpenAI 的一般建议是：先最大化单个 Agent 的能力。更多 Agent 可以带来直观的概念拆分，但也会增加复杂度和开销；很多场景下，一个带工具的 Agent 已经足够。

当复杂工作流中的指令和工具被拆分到多个 Agent 后，性能和可扩展性可能提升。如果 Agent 经常无法遵循复杂指令，或者经常选错工具，就可能需要进一步拆分系统。

拆分 Agent 的实用判断：

- **复杂逻辑**：prompt 中有大量 if-then-else 分支，模板难以扩展，可以把不同逻辑片段拆给不同 Agent。
- **工具过载**：问题不只是工具数量，而是工具相似或重叠。明确工具名、参数和描述后仍无法改善时，可以考虑多个 Agent。

#### 多 Agent 系统

多 Agent 系统可以按具体工作流设计成许多形态。OpenAI 总结了两类常见模式：

1. **Manager 模式，也叫 agents as tools**：中心 manager Agent 通过工具调用协调多个专门 Agent。
2. **去中心化模式，也叫 agents handing off to agents**：多个 Agent 作为同级节点，按专业分工互相 handoff。

可以把多 Agent 系统建模成图：Agent 是节点。Manager 模式中，边表示工具调用；去中心化模式中，边表示执行权转移。

无论采用哪种编排方式，原则都相同：组件保持灵活、可组合，并由清晰结构化的 prompt 驱动。

##### Manager 模式

Manager 模式让一个中心 LLM 负责协调专门 Agent 网络。Manager 在适当时机把任务委派给正确 Agent，再综合结果，形成统一的用户体验。

这个模式适合你希望只有一个 Agent 控制工作流执行并直接接触用户的场景。

##### 声明式图与非声明式图

有些框架要求开发者预先用图声明所有分支、循环和条件。这样有视觉清晰度，但当工作流变得动态复杂时，会变得笨重，并可能要求学习专门 DSL。

Agents SDK 采用更灵活的 code-first 方式。开发者可以用熟悉的编程结构直接表达工作流逻辑，不需要一开始就预定义完整图。

#### 去中心化模式

去中心化模式中，Agent 可以把工作流执行权 handoff 给另一个 Agent。handoff 是单向转移：一个 Agent 调用 handoff 函数后，执行会切到新 Agent，并转移最新对话状态。

这种模式让多个 Agent 处于同等地位。它适合不需要单一中心 Agent 维持控制或汇总结果的场景，而是让专门 Agent 接管任务并直接与用户交互。

这种模式特别适合对话分流，也适合你希望专门 Agent 完全接管某些任务，而原 Agent 不必继续参与的场景。必要时，第二个 Agent 也可以配置 handoff 回原 Agent。

## 护栏

设计良好的护栏可以帮助管理数据隐私风险，例如防止系统提示词泄露；也可以管理声誉风险，例如确保模型行为符合品牌要求。

护栏是所有 LLM 部署的重要组成部分，但它不能替代认证、授权、访问控制和标准软件安全措施。

可以把护栏看成分层防御。单个护栏通常不够，多种专门护栏组合起来，才能让 Agent 更稳健。

### 护栏类型

##### 相关性分类器

确保 Agent 的响应保持在预期范围内，并标记离题输入。

##### 安全分类器

检测越狱和 prompt injection 等不安全输入，防止用户试图利用系统漏洞。

##### PII 过滤器

检查模型输出，避免不必要地暴露个人身份信息。

##### Moderation

标记仇恨、骚扰、暴力等有害或不合适输入，维持安全、尊重的交互。

##### 工具保护

根据工具风险给工具打低、中、高风险等级。评估因素包括只读还是写入、是否可逆、所需权限、财务影响等。高风险工具可以触发额外检查、暂停执行或升级人工。

##### 规则型保护

使用黑名单、输入长度限制、正则过滤等确定性措施，防御已知威胁，例如禁用词或 SQL 注入。

##### 输出验证

通过 prompt engineering 和内容检查，确保输出符合品牌价值，避免损害品牌完整性。

### 构建护栏

构建护栏应从已知风险开始，再根据真实边界情况和失败案例逐步增加。

OpenAI 给出的经验法则：

1. 优先关注数据隐私和内容安全。
2. 根据真实边界情况和失败案例添加新护栏。
3. 同时优化安全性和用户体验，随着 Agent 演进调整护栏。

> 代码示例位置：原文此处展示 Agents SDK 中使用 input guardrail、guardrail output 和 tripwire 的示例。

Agents SDK 把护栏作为一等概念。默认采用乐观执行：主 Agent 先主动生成输出，护栏并发运行；如果违反约束，就触发异常。

护栏可以实现为函数，也可以实现为 Agent，用于执行越狱防护、相关性验证、关键词过滤、黑名单、防护分类等策略。

##### 规划人工介入

人工介入是重要安全机制，尤其适合部署早期。它能帮助发现失败、挖掘边界情况，并建立更稳健的评估循环。

当 Agent 无法完成任务时，应能优雅地把控制权交给人类。在客服场景中，就是升级给人工客服；在 coding agent 场景中，就是把控制权还给用户。

通常需要人工介入的触发条件：

- **超过失败阈值**：限制重试次数或行动次数。超过限制就升级人工。
- **高风险动作**：敏感、不可逆、高影响的动作需要人工监督，例如取消订单、批准大额退款、付款。

## 结论

Agent 标志着工作流自动化进入新阶段：系统可以在模糊场景中推理，跨工具执行动作，并以较高自主性处理多步骤任务。

和更简单的 LLM 应用不同，Agent 会端到端执行工作流，因此适合复杂决策、非结构化数据、脆弱规则系统等场景。

要构建可靠 Agent，需要打好基础：

- 使用能力匹配的模型。
- 定义清楚、可测试的工具。
- 编写结构化指令。
- 根据复杂度选择编排模式，先从单 Agent 开始，必要时再演进到多 Agent。
- 在输入过滤、工具使用、人工介入等层面加入护栏。

成功部署不是“一步到位”。更稳妥的路径是从小范围开始，用真实用户验证，再逐步扩展能力。用正确基础和迭代方式，Agent 可以创造实际业务价值：不仅自动化任务，也智能、适应性地自动化完整工作流。
