## 页面摘要

Anthropic 根据大量 LLM Agent 项目经验指出：有效的 Agent 系统通常依赖简单、可组合、易调试的模式，而不是复杂框架或过度设计。

---

## 引言

过去一段时间，Anthropic 和许多行业团队一起构建 LLM Agent。他们观察到，成功项目往往不是靠复杂框架取胜，而是靠清晰、简单、可组合的工程模式。

这篇文章的目标是总结这些实践经验，并给开发者一套判断和构建 Agentic System 的实用框架。

---

## 什么是 Agent？

“Agent”这个词在不同团队里含义并不完全一致。有些人把它理解为可以长时间自主工作的系统，有些人则把固定流程式的 LLM 应用也称为 Agent。

Anthropic 使用更宽泛的概念：**Agentic Systems**。在这个大类下面，他们进一步区分两种架构：

- **Workflow**：LLM 和工具由开发者写好的固定代码路径编排。
- **Agent**：LLM 自己决定任务流程、工具调用和下一步行动。

这个区分很关键。Workflow 更偏可预测、可控制；Agent 更偏动态、自主和开放。

---

## 什么时候该用 Agent，什么时候不该用？

构建 LLM 应用时，Anthropic 的建议是先从最简单方案开始。很多任务不需要 Agent，也不需要复杂的多步系统。

更复杂的 Agentic System 会带来代价：

- 延迟更高。
- 调用成本更高。
- 调试更难。
- 错误可能在多轮步骤中累积。

如果任务路径清晰、步骤稳定，Workflow 往往比 Agent 更合适。如果任务开放、步骤不可预知、需要模型持续决策，才更适合考虑 Agent。

对于很多应用来说，优化单次 LLM 调用、加入检索、加入上下文示例，已经能达到足够好的效果。

---

## 什么时候以及如何使用框架？

市面上有很多能帮助构建 Agentic System 的框架和平台，例如：

- Claude Agent SDK。
- AWS 的 Strands Agents SDK。
- Rivet 这类可视化 LLM Workflow 工具。
- Vellum 这类用于构建和测试复杂 Workflow 的平台。

这些框架能简化模型调用、工具定义、响应解析和调用链编排，适合快速开始。

但框架也会引入额外抽象层。抽象层太厚时，开发者可能看不清底层 prompt、模型响应和工具调用过程，调试会更困难，也容易误以为系统做了某些其实没有做的事情。

Anthropic 的建议是：先理解直接使用 LLM API 的基础模式。很多模式用少量代码就能实现。如果使用框架，也要理解框架底层的真实行为。

## 构建块、Workflow 和 Agent

这一部分按复杂度从低到高介绍常见模式：

1. 增强型 LLM。
2. 多种 Workflow 模式。
3. 更开放的 Agent。

这些模式不是固定模板，而是可以组合、裁剪、替换的工程积木。

---

## 构建块：增强型 LLM

Agentic System 的基础是增强型 LLM，也就是在普通模型调用上加入能力扩展：

- 检索。
- 工具。
- 记忆。

现代模型可以主动使用这些能力，例如自己生成搜索查询、选择工具、判断需要保留哪些信息。

真正重要的不是“有没有工具”，而是工具是否适合具体场景，接口是否清楚，说明是否足够完整，模型是否容易正确使用。

## Workflow：Prompt Chaining

Prompt Chaining 是把任务拆成一连串固定步骤。每一步 LLM 调用处理上一步的输出，中间可以加入程序化检查，判断流程是否仍然符合预期。

### 适用场景

当任务可以被清楚拆成固定子任务时，这种模式很合适。它用更多步骤换取更高准确率，因为每个 LLM 调用只需要解决一个更小的问题。

### 示例

- 先生成营销文案，再翻译成另一种语言。
- 先写文档大纲，检查大纲符合要求后，再基于大纲写正文。

## Workflow：Routing

Routing 是先判断输入类型，再把它分发到不同的下游流程、prompt、模型或工具。

这种模式可以让不同类别的问题各自优化，避免一个通用 prompt 同时处理所有情况而互相干扰。

### 适用场景

当输入可以被可靠分类，而且不同类别适合不同处理方式时，Routing 很有用。

### 示例

- 客服系统中，把普通问题、退款请求、技术支持分给不同流程。
- 简单问题交给低成本模型，复杂问题交给更强模型。

## Workflow：Parallelization

Parallelization 是让多个 LLM 调用并行工作，再用程序把结果合并。

常见形式有两种：

- **Sectioning**：把任务拆成互相独立的部分并行处理。
- **Voting**：让多个调用处理同一任务，再根据投票或聚合规则得到结果。

### 适用场景

当子任务彼此独立、可以并行提速，或者需要多个角度提高置信度时，可以考虑并行化。

### 示例

- 一个模型处理用户请求，另一个模型同时做安全检查。
- 多个评估 prompt 分别检查模型输出的不同方面。
- 多个安全审查 prompt 从不同角度检查代码问题。
- 内容审核中，不同 prompt 针对不同风险维度投票。

## Workflow：Orchestrator-Workers

Orchestrator-Workers 模式中，一个中心 LLM 负责分析任务、拆分子任务、分配给多个 worker LLM，最后再汇总结果。

### 适用场景

当任务复杂，而且无法提前知道要拆成哪些子任务时，这种模式更合适。

例如代码修改任务中，需要改哪些文件、每个文件如何改，通常要根据具体任务动态决定。

### 示例

- 需要跨多个文件修改的 coding 产品。
- 需要从多个来源查找、筛选、分析信息的搜索任务。

> 图示位置：原文此处展示中心编排者和多个 worker。

## Workflow：Evaluator-Optimizer

Evaluator-Optimizer 模式中，一个 LLM 先生成结果，另一个 LLM 负责评估和反馈，然后进入迭代优化循环。

### 适用场景

这种模式适合有明确评价标准，而且迭代确实能带来质量提升的任务。

一个判断信号是：人类能清楚说出输出哪里需要改进；另一个信号是：模型能根据反馈有效改进结果。

### 示例

- 文学翻译中，评估者指出语气、细节、表达上的问题，再由生成者修改。
- 复杂搜索中，评估者判断信息是否足够，是否需要继续搜索。

## Agent

随着模型在理解复杂输入、推理规划、可靠使用工具、从错误中恢复等方面变强，Agent 开始进入生产实践。

Agent 通常从用户命令或交互式讨论开始。它会自己规划任务，在环境中调用工具，读取反馈，再决定下一步。这个循环可能持续多轮，直到任务完成或需要人类介入。

Agent 的实现不一定复杂。核心常常就是：

1. LLM 获取任务和环境状态。
2. LLM 选择下一步和工具。
3. 系统执行工具。
4. LLM 读取反馈。
5. 循环继续。

### 适用场景

当问题开放、步骤数不可预测、很难硬编码固定路径时，Agent 才更有价值。

但 Agent 成本更高，也更容易因为多轮决策产生累积错误。因此需要：

- 沙盒环境。
- 充分测试。
- 合理护栏。
- 清晰的工具设计。

### 示例

- 解决 SWE-bench 任务的 coding Agent。
- 能使用电脑完成任务的 computer use Agent。

## 组合和定制这些模式

这些模式不是必须照搬的架构。开发者可以根据业务需求组合它们。

关键不是“系统看起来多智能”，而是持续测量效果，并根据评估结果迭代。

Anthropic 强调：只有当更复杂的系统确实带来更好结果时，才值得增加复杂度。

---

## 总结

在 LLM 应用中，成功不是构建最复杂的系统，而是构建适合需求的系统。

推荐路径是：

1. 从简单 prompt 开始。
2. 用全面评估优化 prompt。
3. 当简单方案不足时，再加入多步骤 Workflow 或 Agent。

实现 Agent 时，Anthropic 建议遵循三条原则：

- **保持设计简单**：避免不必要的复杂流程和抽象。
- **保持过程透明**：明确展示 Agent 的计划和步骤。
- **认真设计 ACI**：把 Agent 使用工具和环境的接口当成重要产品来打磨。

框架可以帮助快速起步，但生产环境中不要害怕减少抽象层，回到更基础、可理解、可调试的组件。

## 附录 1：实践中的 Agent

Anthropic 在客户案例中看到两个特别有价值的应用方向：客服和 coding agent。

这两类任务都具有一个共同点：既需要自然语言交互，也需要工具调用和可验证的行动结果。

---

## A. 客户支持

客户支持很适合从普通 chatbot 演进为更强的 Agent，因为它天然需要对话、信息查询和业务动作。

这类系统通常需要：

- 读取客户数据。
- 查询订单历史。
- 检索知识库文章。
- 执行退款、更新工单等操作。
- 以“问题是否解决”作为效果衡量标准。

一些公司已经采用按成功解决问题收费的模式，这说明它们对 Agent 的实际效果有一定信心。

---

## B. Coding Agents

软件开发是 LLM Agent 很有潜力的领域。能力已经从代码补全，逐步扩展到自主解决问题。

Coding Agent 有几个优势：

- 代码结果可以通过测试验证。
- Agent 可以根据测试反馈继续迭代。
- 问题空间相对结构化。
- 输出质量更容易客观评估。

Anthropic 提到，他们的 Agent 已经可以根据 PR 描述解决 SWE-bench Verified 中的真实 GitHub 问题。

不过，自动测试只能验证一部分正确性。代码是否符合系统设计、是否可维护、是否真正满足需求，仍然需要人类审查。

---

## 附录 2：为工具做 Prompt Engineering

无论构建 Workflow 还是 Agent，工具通常都是关键部分。

工具让模型可以访问外部系统和 API。模型会根据工具定义决定是否调用工具、如何传参、如何处理返回结果。

同一个动作可以有多种工具设计方式。例如编辑文件时，可以让模型写 diff，也可以让它重写整个文件。结构化输出可以放在 Markdown 里，也可以放在 JSON 里。

这些格式选择会明显影响模型表现。

### 工具格式设计建议

- 给模型足够空间先思考，不要让它过早陷入难以修正的格式。
- 尽量使用模型在自然文本中熟悉的格式。
- 减少格式负担，例如复杂转义、精确计数、脆弱的行号定位。

### ACI：Agent-computer interface

Anthropic 建议像重视 HCI 一样重视 ACI。也就是说，既然人机界面需要认真设计，Agent 使用工具的界面也需要认真设计。

可以从这些问题检查工具设计：

- 模型只看工具名、描述和参数，能否自然理解怎么用？
- 参数名是否清楚？
- 描述是否像高质量 docstring？
- 是否提供了足够示例？
- 多个类似工具之间是否容易混淆？
- 是否通过测试发现了模型常见误用？
- 是否能通过参数设计减少出错机会？

原文提到，在构建 SWE-bench Agent 时，工具优化花费的时间甚至超过整体 prompt 优化。一个例子是：相对路径容易在工作目录变化后导致模型出错，因此工具接口需要避免这类高频错误。
