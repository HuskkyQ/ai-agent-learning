# 《A Practical Guide to Building AI Agents》中文解析

> 来源：OpenAI Business  
> 原文：[A practical guide to building agents](https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/)  
> 抓取日期：2026-07-06  
> 文档类型：中文解析与学习笔记

## 一句话总结

这篇 OpenAI 指南的核心观点是：Agent 不是“更会聊天的模型”，而是能在明确指令、工具和护栏下，代表用户端到端执行工作流的系统；构建时应从单 Agent、清晰工具和可评估流程开始，再按复杂度逐步演进到多 Agent。

## 文章结构

文章按照“定义 -> 适用场景 -> 设计基础 -> 编排模式 -> 护栏 -> 部署建议”的顺序展开。

可以把它看成一套 Agent 产品落地框架：

1. 先判断什么才算 Agent。
2. 再判断这个问题是否值得用 Agent。
3. 然后设计模型、工具、指令。
4. 接着选择单 Agent 或多 Agent 编排。
5. 最后用护栏和人工介入降低生产风险。

## Agent 的定义重点

OpenAI 对 Agent 的定义比较工程化：Agent 是能代表用户独立完成任务的系统。

它不是单纯“会回复”的聊天机器人，而是具备两个能力：

- LLM 能控制工作流执行，决定下一步、判断完成、处理失败。
- Agent 能调用工具获取信息或执行动作，并在护栏内运行。

这个定义很适合你 Day 1 的学习问题：Agent 与 chatbot 的区别不在对话界面，而在“是否控制并执行工作流”。

## 什么时候值得做 Agent

文章给出的判断标准很实用：不要因为 Agent 很热门就默认使用它。Agent 适合传统自动化难以覆盖的场景。

更适合 Agent 的任务有三类：

- 需要复杂判断，规则很难穷举。
- 规则系统已经庞大、难维护、容易出错。
- 输入主要是非结构化数据，例如自然语言、文档、对话。

不适合 Agent 的情况：

- 规则简单且稳定。
- 工作流可完全确定。
- 单次 LLM 调用或普通 RAG 已经足够。
- 风险高但没有成熟护栏和人工介入机制。

## 三个设计基础

OpenAI 把 Agent 设计拆成三个基础件：模型、工具、指令。

### 1. 模型

模型选择不是“永远用最强模型”，而是先用强模型建立上限，再通过 evals 判断哪些步骤可以换成更小、更便宜、更快的模型。

这对应一个工程原则：先验证能力，再优化成本。

### 2. 工具

工具是 Agent 能从“会说”变成“会做”的关键。

工具分为三类：

- 数据工具：读信息，例如查数据库、读 PDF、搜网页。
- 动作工具：写入或执行，例如发邮件、更新 CRM、转人工。
- 编排工具：把其他 Agent 当成工具调用。

工具设计的重点不是数量，而是清晰度、可发现性、可测试性和可复用性。

### 3. 指令

Agent 的指令需要比普通 chatbot prompt 更像操作规程。

好的指令应该：

- 来自真实 SOP、政策、支持脚本。
- 拆成明确步骤。
- 每一步对应清楚动作或输出。
- 覆盖常见边界情况。

这意味着构建 Agent 往往不是只写 prompt，而是把业务流程转成模型可执行的流程规范。

## 编排：先单 Agent，再多 Agent

这篇文章的一个重要建议是：先最大化单 Agent 能力，不要过早上多 Agent。

单 Agent 的优点：

- 复杂度低。
- 评估更简单。
- 维护更容易。
- 工具和指令的行为更容易定位问题。

当以下问题反复出现时，再考虑多 Agent：

- 单个 prompt 里条件分支太多。
- 工具数量或相似度导致模型经常选错工具。
- 不同任务领域需要明显不同的指令、上下文或权限。

## 多 Agent 的两种模式

### Manager 模式

一个中心 Agent 负责和用户交互，也负责调度多个专门 Agent。专门 Agent 对 manager 来说像工具。

适合场景：

- 需要统一用户体验。
- 需要中心 Agent 汇总结果。
- 希望一个 Agent 控制整体工作流。

典型例子：用户要求多语言翻译，manager 分别调用西班牙语、法语、意大利语 Agent，再汇总返回。

### 去中心化 Handoff 模式

多个 Agent 平级协作，一个 Agent 可以把执行权交给另一个 Agent。

适合场景：

- 对话分流。
- 专门 Agent 接管后可以独立完成任务。
- 不需要中心 Agent 持续汇总。

典型例子：客服 triage Agent 判断用户问的是订单问题，然后把会话交给订单管理 Agent。

## 护栏是生产级 Agent 的核心

文章把 guardrails 放在非常重要的位置。Agent 会调用工具、执行动作、处理真实用户输入，因此风险比普通问答系统更高。

护栏不是单一机制，而是分层防御：

- 相关性分类器：防止跑题。
- 安全分类器：防 prompt injection 和越狱。
- PII 过滤：避免泄露个人信息。
- Moderation：过滤有害内容。
- 工具保护：按风险控制工具调用。
- 规则型保护：黑名单、长度限制、正则等确定性防护。
- 输出验证：确保输出符合品牌和业务要求。

真正生产可用的 Agent 还必须配合认证、授权、访问控制和普通软件安全措施。

## 人工介入不是失败，而是设计

OpenAI 明确建议规划 human-in-the-loop。尤其在部署早期，人工介入能帮助发现失败模式、补充评估集、提升系统可靠性。

应该触发人工介入的情况：

- Agent 超过重试或行动次数。
- 用户意图多次识别失败。
- 动作敏感、不可逆或高风险。
- 涉及财务、账户、订单取消、大额退款等场景。

这个观点对实践很重要：生产级 Agent 不是“完全不需要人”，而是知道什么时候应该把控制权交还给人。

## 和 Anthropic 那篇文章的关系

这篇 OpenAI 指南和 Anthropic 的《Building effective agents》有高度一致的工程思想：

- 都强调不要为了复杂而复杂。
- 都建议从简单模式开始。
- 都认为工具设计是 Agent 成败关键。
- 都把 workflow/agent 的边界放在“谁控制流程”上。
- 都强调评估、可调试性和护栏。

差异在于：

- Anthropic 更强调常见 workflow 模式，例如 prompt chaining、routing、parallelization、evaluator-optimizer。
- OpenAI 更偏产品落地路径，重点讲模型、工具、指令、编排、护栏、人工介入。

两篇合起来，可以形成完整学习框架：Anthropic 帮你理解模式，OpenAI 帮你理解落地组件。

## 对学习 Agent 开发的启发

如果你要按学习路线实践，可以把这篇文章拆成四个练习：

1. **定义练习**：用自己的话解释 chatbot、workflow、agent 的区别。
2. **用例判断练习**：找 3 个业务流程，判断哪些适合 Agent，哪些不适合。
3. **工具设计练习**：为一个客服 Agent 设计 5 个工具，并标注数据工具、动作工具、编排工具。
4. **护栏设计练习**：为一个退款 Agent 设计输入过滤、工具风险等级、人工介入条件。

## 可执行 Checklist

构建第一个 Agent 前，可以逐项检查：

- 我是否清楚定义了用户目标和完整 workflow？
- 这个任务是否真的需要 Agent，而不是规则或单次 LLM？
- 是否有 evals 来衡量准确率、成本、延迟和失败率？
- 是否先用强模型建立性能基线？
- 每个工具是否有清晰名称、参数、描述和测试样例？
- 指令是否来自真实 SOP 或政策，而不是随手写的 prompt？
- 是否优先尝试单 Agent？
- 什么时候才会拆成多 Agent？
- 是否有防 prompt injection、PII、越权工具调用的护栏？
- 哪些动作必须人工确认？

## 我的理解

这篇文章的核心不是教你写一个炫酷 demo，而是教你把 Agent 当成一个生产系统来设计。

一个 Agent 项目真正困难的部分通常不是“调用模型”，而是：

- 把业务流程表达清楚。
- 把工具设计清楚。
- 把失败路径设计清楚。
- 把评估和护栏做好。
- 在成本、延迟、准确率、用户体验之间取舍。

所以学习 Agent 开发时，不应该只关注框架 API。更重要的是掌握 workflow 拆解、工具接口设计、评估集构建、护栏和人工介入这些工程能力。

## 下一步建议

建议基于这篇文章做一个小项目：

> 构建一个“订单查询客服 Agent”设计稿，不一定先写代码。

输出内容：

1. 用户目标。
2. 工作流步骤。
3. 需要的工具列表。
4. 单 Agent 还是多 Agent。
5. 指令草稿。
6. 护栏列表。
7. 人工介入条件。

这会比直接套框架更能训练 Agent 产品和工程思维。
