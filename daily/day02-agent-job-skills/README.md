我的理解（自由记录）：

- Agent 开发岗更偏应用工程，不是从零训练大模型。核心工作是把 LLM 放进真实业务流程里，让它能理解任务、调用工具、检索知识、执行步骤、处理失败，并用评估和部署手段保证效果稳定。
- 大模型算法岗更关注模型本身，例如预训练、微调、对齐、推理优化、模型结构和训练数据；Agent 应用开发岗更关注模型如何被产品和业务可靠地使用。

更完整回答：

1. 基础编码能力：Python / TypeScript、HTTP API、异步任务、错误处理、日志、测试。这是把 Agent 做成可运行系统的基础。
2. LLM API 能力：理解 messages、system/user/assistant 角色、token、上下文窗口、流式输出、结构化输出和模型参数。没有这层，就很难稳定调用模型。
3. Prompt 与任务拆解：把模糊目标拆成清晰步骤，设计输入输出格式，控制模型行为边界。这是 Agent 能不能正确思考和执行的起点。
4. Tool Calling：设计工具接口、参数 schema、返回格式和失败处理，让模型能调用搜索、数据库、文件、代码执行、业务系统等外部能力。
5. RAG 与知识库：把外部文档、业务资料、历史记录检索出来交给模型使用，解决模型不知道、记不住、容易编造的问题。
6. Workflow / Orchestration：把多步骤任务组织成稳定流程，决定什么时候用固定流程，什么时候让 Agent 自主决策，什么时候需要人工介入。
7. Eval 与观测：构造测试集、评估准确率/成功率/工具调用质量，记录 trace、日志和失败案例，用数据判断 Agent 是否真的变好。
8. Deploy 与工程化：把 Agent 部署成服务或产品功能，处理环境变量、权限、安全、限流、成本、监控和线上回滚。

当前最需要补强的 3 项：

1. Prompt 与任务拆解：先学会把任务说清楚、拆清楚，这是后续 Tool、Workflow、Eval 的基础。
2. Tool Calling / Workflow：Agent 和普通聊天机器人的关键区别在于能调用工具并完成多步骤任务。
3. Eval：如果不会评估，就无法判断 Agent 是否可靠，也无法持续优化。
